# Operating Model

## Zweck
Das Operating Model definiert, wie das persönliche Multi-Projekt-Agentic-Control-System im Alltag arbeitet.

## Betriebsprämissen
- Single-User-System
- Aufmerksamkeit ist die knappste Ressource
- Nicht jede Eingabe wird sofort zu Arbeit
- Nicht jede Arbeit braucht sofort Ausführung
- Nicht jede wiederholte Praxis wird Standard
- Cross-Project-Koordination ist normal
- Das System soll entlasten, nicht unkontrolliert delegieren

## Betriebsmodi
- Intake-Modus
- Planning-Modus
- Execution-Modus
- Review- und Approval-Modus
- Portfolio-Modus
- Learning-Modus

## Hauptbetriebsflüsse
### Neuer Input → System
1. Eingabe trifft ein.
2. Sie bleibt zunächst Interaktionsinput, nicht Arbeit.
3. Semantische Einordnung erfolgt.
4. Das System entscheidet über Einordnung.
5. Erst dann wird Arbeit aufgenommen oder zurückgestellt.

### Neue Idee → Projekt
1. Idee wird als potenzielles Projekt erkannt.
2. Es wird geprüft, ob wirklich ein neues Projekt nötig ist.
3. Bei Bestätigung entsteht ein Candidate.
4. Projektanlage wird initiiert.
5. Projekt wird ins Portfolio aufgenommen.

### Neue Arbeitsanforderung → Work Item
1. Arbeitsabsicht wird bewertet.
2. Dringlichkeit, Duplikat, Projektbezug und Kapazität werden geprüft.
3. Arbeit wird angenommen, verschoben, gebündelt, abgelehnt oder zur Klärung zurückgegeben.
4. Angenommene Arbeit wird als Work Item angelegt.

### Work Item → Plan
1. Work Item wird strukturiert.
2. Ergebnis, Teilaufgaben, Reihenfolge, Dependencies, Gates und Kontext werden identifiziert.
3. Ergebnis ist ein expliziter Arbeitsplan.

### Plan → Ausführung
1. Workflow übernimmt den Plan.
2. Policy/Governance und Portfolio-Lage werden geprüft.
3. Nur dann wird Execution beauftragt.
4. Execution liefert Resultate, Verifikation, Blocker, Evidenz und Artefaktbezüge.

### Lokaler Fehler → kontrollierte Reaktion
1. Fehler wird klassifiziert.
2. Reaktion: Clarify, Wait, Retry, Replan, Reject oder Escalate.
3. Nur wenn intern keine saubere Auflösung möglich ist, wird eskaliert.

### Cross-Project-Abhängigkeit → Steuerung
1. Dependency wird erkannt oder vorgeschlagen.
2. Sie wird als Strukturbeziehung modelliert.
3. Das System bewertet Frei-, Waiting- oder Blocked-Lage.
4. Workflow und Portfolio werden abgestimmt.
5. Entsperrung erfolgt nur auf expliziter Struktur- oder Artefakt-/Evidenzlage.

### Beobachtung → Lernen → Änderung
1. Beobachtungen entstehen aus Arbeit, Fehlern, Entscheidungen und Artefakten.
2. Sie werden klassifiziert und mit Evidenz verknüpft.
3. Manche werden als Kandidaten markiert.
4. Kandidaten werden geprüft.
5. Nur geeignete Kandidaten werden zu Standards.
6. Nur ein Teil der Standards wird verbindlich gebunden.
